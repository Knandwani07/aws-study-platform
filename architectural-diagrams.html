AWS EKS Cloud Infrastructure Overview

This diagram shows how a secure and scalable cloud infrastructure is built using Amazon EKS (Elastic Kubernetes Service) to deploy and manage containerized applications.

Infrastructure as Code (IaC):

The setup begins with Terraform, which automates the creation and management of all AWS resources. GitHub Actions is used for continuous integration and deployment, ensuring that every infrastructure update happens smoothly and consistently.

AWS Cloud Environment:

The environment is divided into two Availability Zones to ensure high availability and fault tolerance. Each zone contains a Virtual Private Cloud (VPC) with public and private subnets.

Public Subnets:

These subnets host resources that need internet access, such as the EKS Public Load Balancer and NAT Gateway, which route external and internal traffic securely.

Private Subnets:

These subnets host worker nodes, where actual applications run.

  • External apps handle requests from the internet.
  • Internal apps serve communication within the organization or other private services.
  • The EKS Private Load Balancer distributes incoming traffic among these applications for better performance and reliability.

Networking and Connectivity:

The Internet Gateway enables secure public access, while the VPN Gateway and AWS Direct Connect link the AWS environment to the corporate data center, allowing hybrid cloud communication.

App and Data Management:

Supporting services like Amazon Route 53, Secrets Manager, KMS, ECR, RDS, EFS, and S3 help with DNS management, security, image storage, databases, file systems, and data storage.

Security and Access:

The infrastructure includes separate inbound and outbound paths for internal and external communication to maintain strong network isolation and protect applications.

Overall, this architecture ensures a secure, automated, and highly available cloud environment for running applications at scale using Kubernetes on AWS.

Amazon EKS Architecture: Fargate and EC2 Managed Node Groups

This diagram illustrates how Amazon Elastic Kubernetes Service (EKS) can run containerized applications using two different compute options, AWS Fargate and Amazon EC2 Managed Node Groups.

Amazon EKS (Control Plane):

At the center, EKS acts as the control plane that manages and orchestrates containerized workloads (pods). It decides where and how these workloads should run across the available compute resources.

Serverless Container Data Plane (AWS Fargate):

On the left, Fargate provides a serverless way to run containers. It automatically provisions and scales the compute resources required to run your Kubernetes pods. This means you do not need to manage servers or infrastructure — AWS handles it for you.

Managed Node Group EC2 Data Plane:

On the right, the setup uses Amazon EC2 instances grouped into Managed Node Groups. These instances run your pods within your own virtual private cloud (VPC). EC2 Auto Scaling ensures that capacity adjusts automatically based on workload demand.

The diagram also shows multiple Availability Zones to provide high availability and fault tolerance.

Networking and Scaling:

Both Fargate and EC2 environments operate within their respective VPCs, ensuring isolation and security. Fargate scales automatically, while EC2 nodes use Auto Scaling groups to adjust capacity dynamically.

In summary, this architecture demonstrates a flexible and scalable way to run Kubernetes workloads on AWS, either through the fully managed, serverless Fargate platform or with customizable EC2-managed node groups.

CI/CD Pipeline for Game App

An automated pipeline that takes code from a developer, builds and scans a container image, deploys it to a managed Kubernetes service (EKS Fargate), and delivers the game to users through an AWS load balancer.

How it works (step-by-step):

  1. Developer checks in code: A developer writes code and pushes it to a GitHub repository.
  2. Automated build and test: GitHub Actions automatically runs the build and test workflows when code is pushed.
  3. Create a Docker image: The application is packaged into a Docker image that contains everything needed to run it.
  4. Security scan: The image is scanned with a tool (Trivy) to detect known vulnerabilities.
  5. Store the image: After passing the scan, the image is pushed to Docker Hub (a central image registry).
  6. Deploy to cloud: The verified image is deployed to Amazon EKS running on AWS Fargate, which runs containers without you managing servers.
  7. Serve users: An AWS Load Balancer distributes incoming traffic to the application so end users can access the game reliably and at scale.

Why this setup matters:

  • Automation: Code moves from commit to production automatically, reducing manual steps.
  • Security: Images are scanned before deployment to catch vulnerabilities early.
  • Scalability: EKS with Fargate and the load balancer let the application handle varying user loads.
  • Speed: Faster, repeatable releases make it easier to deliver updates and fixes.
